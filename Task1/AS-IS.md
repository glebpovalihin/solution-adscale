# AS-IS анализ архитектуры системы AdScale

## 1. Общая информация о системе

**Компания:** AdScale — SaaS-платформа для размещения цифровой рекламы с поддержкой RTB (Real-Time Bidding).
**Текущая нагрузка:** до 15 000 RPS в пике.
**Ключевая проблема:** система работает на пределе — P95 latency 110–120 ms при требовании не более 100 ms, периодические деградации в часы пик, отсутствие запаса производительности.
**Бизнес-контекст:** интеграция с крупной DSP-площадкой добавит +20% трафика немедленно и двукратный рост через год. Новый партнёр требует SLA с latency не более 80 ms и штрафы за деградацию/простои.

## 2. Архитектура

Система представляет собой **гибридный монолит**:
- на стеке C++ / Go / Python / Node.js
- развёрнутый на одном production-кластере
- с единой инсталяцией PostgreSQL
- без контейнеризации, кеширования и очередей
- деплой — ручной.

## 3. База данных

- база данных — одна PostgreSQL
- все модули работают с общей схемой
- риск блокировок - тяжелые аналитические запросы (Analytics Service) могут блокировать строки увеличивая latency не связанных компонентов
- нет read-replicas — все компоненты работают с одной БД на запись и чтение то приводит к конкуренции

## 4. Кеширование

- отсутствие кеширования приводит к постоянному чтению из БД, увеличивая нагрузку на PostgreSQL и latency

## 5. Очереди

- отсутствие очередей приводит к тому что все операции синхронны, что в свою очередь увеличивает связность компонентов системы
- нет возможности отделения не критичных операций/процессов

### 6. Автоматизации
- деплой вручную — высокий риск человеческой ошибки
- невозможность быстрого независимого (в ряде случаев автоматического) масштабирования компонентов
- нет оркестрации
- один production-кластер, недоступность которого приводит к полной недоступности всей системы

## 7. Риски

- деградация lantency сверх допустимых значений (в текущей реализации latency 110–120 ms при требовании в 100 ms на нагрузке меньше ожидаемой - 15 000 rps)
- таймауты синхронных вызовов БД и сбои критичного функционала Auction Engine
- финансовые штрафы при нарушении SLA, репутационные потери, переход DSP к более подготовленным к данной нагруке конкурентам
