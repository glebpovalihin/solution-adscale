# ADR-001: Стратегия эволюции архитектуры (от монолита к микросервисам).

**Дата:** 2026-05-25

**Статус:** Accepted

## Контекст
AdScale — платформа RTB-рекламы, являющаяся гибридным монолитом (C++ / Go / Python / Node.js) с единым хранилищем PostgreSQL.

Текущие показатели:
- 15 000 RPS в пике при пределе системы
- P95 времени отклика 110–120 ms при требовании не более 100 ms
- Периодические деградации в часы пик

Компания получила предложение об интеграции с крупной DSP-площадкой, что даёт:
- +20% к трафику при интеграции, 2х кратный рост через год
- Жёсткий SLA: не более 80 ms времени отклика, штрафы за деградацию и простои
- Пиковая нагрузка до 18 000 RPS через 3 месяца, 50 000 RPS через год

**Архитектурные драйверы:**
- Производительность: P95 не более 100 ms (3 мес.), не более 80 ms (1 год)
- Масштабируемость: горизонтальное масштабирование critical path
- Доступность: 99.9% SLA через год
- Ограничения: бюджет ограничен, срок запуска через 3 месяца, нельзя останавливать бизнес
- Команда: 35 человек (backend-, frontend- и DevOps-разработчики, тестировщики, дизайнеры, сотрудники службы поддержки и менеджмент), нет SRE и выделенного инфраструктурного инженера

Рассматривались четыре альтернативы:
1. Стратегия развития в рамках монолита - точечные оптимизации
2. Переход к модульному монолиту — рефакторинг внутри единого приложения
3. Полный переход на микросервисы — переработка всей системы
4. Strangler Fig Pattern — постепенное выделение сервисов

### Альтернатива 1: Развитие в монолите
Плюсы:
- нулевые затраты на миграцию.

Минусы:
- система уже на пределе (15K RPS)
- единая PostgreSQL — SPOF и конкурентный доступ
- нет изоляции critical path
- невозможно горизонтальное масштабирование
- P95 уже превышает 100ms — интеграция с DSP только ухудшит ситуацию

### Альтернатива 2: Модульный монолит
Плюсы:
- выраженные границы модулей
- изолированные схемы данных
- сохранение ACID
- готовность к будущей декомпозиции

Минусы:
- невозможность независимого масштабирования критичного по latency Auction Engine
- единая PostgreSQL остаётся bottleneck'ом
- не решает проблему конкуренции OLTP и аналитики

### Альтернатива 3: Полный переход на микросервисы
Плюсы:
- подходящий архитектурный стиль [согласно факторам](../Drivers.md)
- независимое масштабирование
- полная изоляция

Минусы:
- жесткий дедлайн запуска не позволяет инвестировать время в переписывание всей системы с нуля без остановки бизнеса
- риск потери ACID-транзакций
- команда не имеет опыта разработки микросервисной инфраструктуры
- высокий риск срыва сроков и downtime

### Альтернатива 4: Strangler Fig Pattern (постепенное выделение сервисов)
Плюсы:
- поэтапная миграция без остановки бизнеса
- бизнес-ценность с первого шага
- возможность инкрементально улучшать SLA
- снижение рисков за счёт итеративности

Минусы:
- временное переходное состояние логики монолита и микросервисов
- необходимость поддерживать интеграционные адаптеры
- требуется дисциплина и осознанность при выделении bounded context'ов

## Решение

Вариант 4. **Strangler Fig Pattern** — постепенная эволюция от монолита к микросервисной архитектуре с событийно-ориентированной интеграцией в несколько этапов.

### Этап 1 (0–3 месяца): Выделение RTB Service
- RTB Service выносится в отдельный микросервис (C++/Go) — так как является наиболее latency-критичный компонентом
- Добавляется слой Redis-кеша для RTB Service
- RTB Service интегрирует собственную PostgreSQL (Primary + Read Replica)
- Добавляется Kafka для асинхронного обновления ставок (топик `ставки`)
- Монолит продолжает работать, но делегирует аукцион новому сервису, текущая логика постепенно модулизируется

Цель: снизить P95 до 100 ms, выдержать 18 000 RPS, подключить DSP-партнёра

### Этап 2 (3–6 месяцев): Выделение AD Server, Campaign Service и Statistic Service
- Реализация API Gateway, DSP Gateway, Advertiser Gateway
- Kafka-топики: `кампании`, `таргетинг`, `показы`, `клики`
- AD Server (Go) - маршрутизация bid запросов, оптимизированный подбор кандидатов на основе кеш слоя
- Campaign Service (Go) — CRUD кампаний с собственной PostgreSQL
- Statistic Service (Python) — write-heavy pipeline: Kafka -> ClickHouse балк вставки

Цель: подготовка инфраструктуры для полноценного перехода к микросервисной архитектуры

### Этап 3 (6–12 месяцев): Полная декомпозиция
- Выделение Financial Service (Go) + PostgreSQL
- Выделение Analytics Service (Python) + ClickHouse
- Выделение Delivery Service (Node.js) + PostgreSQL + Redis
- Полностью микросервисная архитектура с event-driven интеграцией
- Независимое горизонтальное масштабирование каждого сервиса

Цель: 50 000 RPS, P95 не более 80 ms, SLA 99.9%

## Последствия
**Положительные:**
- Бизнес-ценность с первого этапа — DSP-партнёр подключается через 3 месяца
- Нулевой downtime — монолит замещается постепенно (Strangler Fig)
- Каждый этап даёт измеримое улучшение: снижение latency, рост RPS
- Команда учится микросервисам постепенно
- Явно выделенные bounded context'ы с собственной БД устраняют конкуренцию за ресурсы
- Асинхронные очереди (Kafka) буферизируют нагрузку и не добавляют дополнительной связности в сервисы
- Горизонтальное масштабирование становится возможным для каждого сервиса изолированно

**Отрицательные:**
- Временное сосуществование монолита и микросервисов (первые 6–12 месяцев)
- Накладные расходы на сетевые вызовы между сервисами (частично компенсируются кэшем и gRPC)
- Необходимость поддерживать два типа деплоя (монолит + микросервисы) на этапе миграции
- Требуется внедрение CI/CD, контейнеризации и оркестрации

**Риски:**
- Если на Этапе 1 RTB Service не уложится в 80 ms — штрафы DSP уже с первого месяца
- При неправильном выделении границ bounded context'а может потребоваться перепроектирование
- Команда без SRE может испытывать трудности с эксплуатацией Kafka и Redis
- Временное дублирование данных между монолитом и микросервисами может потребовать синхронизации
