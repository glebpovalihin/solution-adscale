# Масштабирование баз данных

### PostgreSQL

Определяем стандартом наличие репликации Master-Slave для реляционных хранилищ разделяя нагрузку записи и чтения во всех сценариях неравномерного распределения нагрузки:
- RTB System
- Campaign System
- Delivery System

### Caching
Применяем механизмы кеширования для снижения нагрузки на персистентные хранилища и снижения латентности критичного пути:
- RTB System
- Delivery System

### Шардирование

На планируемой нагрузке реализация шардирования яляется избыточной для Postgresql хранилищ, ее внедрение можно рассмотреть по итогам реализации целевого решения по advertiser_id для RTB System.

Для ClickHouse (Statistic System и Analytics System) применяется шардирование по временному ключу (по дням/неделям).
Причины:
- Эффективная вставка time-series данных (каждый шард пишет только свой временной диапазон)
- Параллельную обработку запросов по диапазонам дат
- Возможность охлаждать старые партиции (data retention) без остановки сервиса

# Отказоустойчивость данных

- Запуск в 3х зонах доступности и распределение реплик между всеми зонами
- Регулирование скачков нагрузки через правила HPA систем

### RPO и RTO по сервисам

|Сервис|RPO|RTO|Причины|
|---|---|---|---|
|RTB System|1 мин|10 мин|Критичный путь, проводит аукцион|
|Campaign System|10 мин|1 час|Не критичный путь, настраивает кампании|
|Financial System|10 мин|1 час|Данные можно восстановить из kafka|
|Delivery System|1 мин|10 мин|Критичный путь, отображает креатив|
|Statistic System|1 мин|10 мин|Критичный путь, фиксирует факт отображения и клика|
|Analytics System|1 час|1 час|Данные можно восстановить из kafka|

### Стратегия резервного копирования

Master-slave репликация с возможностью переключения slave реплики в master при недоступности текущего мастера.

### Соответствие принципам 12-факторных приложений

|Принцип|Соответствие|
|---|---|
|Кодовая база|(+) Каждый сервис - свой деплой|
|Зависимости|(+) Docker контейнеризация и декларирование зависимостей|
|Конфигурация|(+) использование переменных окружения (отдельно от кода)|
|Сторонние службы|(+) Подключаются через конфиги, абстракции и заменяются без влияния на приложение|
|Сборка и запуск|(+) наличие CI/CD, строгое разделение стадий, откат через смену тега образа|
|Процессы|(+) Statless микросервисы, состояние в отдельных компонентах (хранилищах), любой процесс может быть остановлен без потери данных|
|Привязка портов|(+) явный binding портов|
|Конкурентные процессы|(+) statless сервисы позволяют запускать N реплик|
|Утилизируемость|(+) statless сервисы позволяют перезапускать процессы независимо|
|Ведение логов|(+) Логи пишутся в стандартный вывод|
|Администрирование|(+) Запуск процессов админиистрирования в виде одноразовых процессов|
|Среды разработки, тестирования и эксплуатации|(+) Docker-образ идентичен во всех средах|
