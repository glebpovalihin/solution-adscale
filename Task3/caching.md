# Дизайн кэширования

Системно используем Redis в качестве сервиса кеширования.

### Что кешируем:
|Система|Что кешируем|TTL|Стратегия инвалидации|
|--|--|--|--|
|RTB System|Активные кампании (active_campaings)|5 мин|Инвалидация при kafka-событиях по кампаниям; по TTL|
|RTB System|Ставки (bid)|1 час|Инвалидация при kafka-событиях по ставке; по TTL|
|RTB System|Таргетинг (targeting)|1 час|Инвалидация при kafka-событиях по таргетингу; по TTL|
|Delivery System|Креативы (creative)|1 час|Инвалидация при изменении креатива в основном хранилище; по TTL|
|Delivery System|Кампании (campaing)|1 час|Инвалидация при kafka-событиях по кампаниям; по TTL|

### Размещение Redis

Собственный Redis-кластер, развёрнутый рядом с инстансами RTB Service и Delivery System (тех же зонах доступности) для минимизации latency.

### Cache Warming

- При старте сервиса:
  - RTB Service загружает в Redis полный список активных кампаний, ставок и правил таргетинга из PostgreSQL.
  - Delivery System загружает в Redis полный список  кампаний и креативов из PostgreSQL
- Через Kafka Consumer: подписка на топики ставок, таргетинга, кампаний — каждое событие обновляет соответствующий ключ в Redis.
- Периодическая синхронизация в RTB Service: фоновый воркер каждые 5 мин сверяет active_campaigns с PostgreSQL для удаления просроченных.
